iOS GCD使用总结

 2017/03/03

GCD全称为Grand Centeral Dispatch，为iOS和OS X的多核硬件上执行提供控制，是异步执行任务的技术之一。
一般将应用程序中记述的线程管理用的代码在系统中的实现。开发者只需要定义想执行的任务并追加到适当的Dispatch Queue中，
GCD就能生成必要的线程并执行计划执行任务。由于线程管理是作为系统的一部分来实现的，因此可以统一管理，也可执行任务，这样就比以前的线程更有效率。
也就是说，GCD用我们难以置信的非常简洁的记述方法，实现了极为复杂繁琐的多线程编程，可以说这是一项划时代的记述。下面是使用了GCD源码的例子，感受
GCD的威力。

Dispatch Queue的种类

虽然有了Dispatch Queue的种类，但如何才能得到Dispatch Queue。

dispatch_queue_create

第一种方法是通过GCD的Api生成Dispatch Queue

Serial Dispatch Queue同时只能执行一个追加处理，但如果将处理分别追加到4个Serial Dispatch Queue中，各个Serial Dispatch Queue执行1个，即为同时执行4个处理
Serial Dispatch Queue只生成并使用一个线程，如果生成2000个Serial Dispatch Queue，那么就生成2000个线程，会消耗大量内存，大幅度降低系统的响应性能。
多个线程更新相同资源导致数据竞争时使用Serial Dispatch Queue

dispatch_queue_create函数的返回值表示Dispatch Queue的”dispatch_queue_t类型”。

Main Dispatch Queue/Global Dispatch Queue

第二种方法是获取系统提供的Dispatch Queue。

第一个参数不可指定为系统提供的Main Dispatch Queue或者Global Dispatch Queue

Dispatch Group

在Dispatch Queue中的多个处理全部结束后想执行结束处理，就可以用的Dispatch Group使代码来的更简单。

在dispatch_group_notify函数中不管制定什么样的Dispatch Queue,属于Dispatch Group的全部处理再追加指定的Block时都已经执行结束。
另外，在Dispatch Group中也可以使用dispatch_group_wait函数仅等待全部处理执行结束。

dispatch_group_wait函数的第二个参数指定为等待时间属于dispatch_time_t类型的值

如果dispatch_group_wait的返回值部位0，就意味着虽然过了指定的时间，但group的某一个处理还在执行中。若果为0，在全部处理结束
当等待时间为DISPATCH_TIME_FOREVER，由group的处理必定全部结束。

dispatch_barrier_async

在访问数据库时，前面说了可以用Serial Dispatch Queue避免数据竞争。写入处理确实不可与其他的写入处理以及包含
读取处理的其他某些处理并行执行。但是如果读取处理只是与读取处理并行执行，就不会有问题了。
也就是说为了高效的访问，读取处理追加到Concurrent Dispatch Queue中，写入处理在任一个读取处理没有执行的状态下
追加到Serial Dispatch Queue中即可.
虽然用Dispatch Group和dispatch_set_target_queue函数可以实现但代码复杂，这里GCD为我们提供了一个简单的方法
dispatch_barrier_async

dispatch_sync

dispatch_async函数的”async”意味着”非同步”，就是将指定的Block”非同步”的追加到指定的Dispatch Queue中。
dispatch_async不作任何等待。
既然有”async”肯定就有”sync”，dispatch_sync就是将指定的Block”同步”追加到指定的Dispatch Queue中，再追加Block结束之前
，dispatch_sync函数会一直等待。

dispatch_apply

dispatch_apply函数是dispatch_sync函数和Dispatch Group的关联Api。该函数按指定的次数将指定的Block追加到指定的Dispatch Queue中，并等待全部处理执行结束

因为在Global Dispatch Queue中处理所以各个处理的执行时间不定。但是Done肯定在最后被输出。

dispatch_suspend / dispatch_resume

当追加大量处理到Dispatch Queue时，再追加处理的过程中，有时希望不执行已追加的处理。例如演算结果被Block截获时
，一些处理会对这个结果造成影响。在这种情况下，只要挂起Dispatch Queue即可，当可执行时在恢复。
dispatch_suspend函数挂起指定的Dispatch Queue

dispatch_resume函数恢复指定的Dispatch Queue

这些函数对已经执行的处理没有影响。挂起后，追加到Dispatch Queue中但尚未执行的处理在此之后停止执行。而恢复则使得这些
处理继续进行

Dispatch Semaphore

当并行执行的处理更新数据时，会产生数据不一致的情况，有事程序还会异常结束。虽然用Serial Dispatch Queue和dispatch_barrier_async函数
可以避免这类问题，但有必要进行更细粒度的排他控制。
例如：不考虑排序，将所有数据追加到NSMutableArray中。

因为使用Global Dispatch Queue更新NSMutableArray对象，所以执行后由内存错误导致程序异常结束额概率很高。
此时应使用Dispatch Semaphore。
Dispatch Semaphore是持有计数的信号，该计数是多线程编程中的计数类型信号。所谓信号，类似于过马路时常用
的手旗。可以通过时举起手旗，不过通过时放下。而在Dispatch Semaphore中，使用计数来实现该功能。计数为0时
等待，计数为1或大于1时，减去1而不等待。

参数表示计数的初始值。本例将计数值初始化为”1”。从函数名称中含有的create可以看出，该函数与Dispatch Queue
和Dispatch Group一样，必须通过dispatch_release函数释放。当然，也可以通过dispatch_retain函数持有。

dispatch_semaphore_wait函数等待Dispatch Semaphore的计数值达到大于或等于1。当计数值大于等于1，或者在待机中
计数值大于等于1时，对该计数进行减法并从dispatch_semaphore_wait函数返回。第二个参数与dispatch_group_wait
函数等相同，由dispatch_time_t类型值指定等待时间。

在没有Serial Dispatch Queue和dispatch_barrier_async函数那么大粒度且一部分处理需要进行排他控制的情况下
，Dispatch Semaphore便可发挥威力。

Dispatch I/O

读取较大文件时，如果将文件分成合适的大小并使用Global Dispatch Queue并列读取的话，应该会比一般的读取速度快不少。
现今的输入/输出硬件已经可以做到一次使用多个线程更快地并列读取了。能实现这一功能的就是Dispatch I/O和Dispatch Data
通过Dispatch I/O读写文件时，使用Global Dispatch Queue将1个文件按某个大小read/write。

可像上面这样，将文件分割为一块一块地进行读取处理。分割读取的数据通过使用Dispatch Data可进行更为简单地进行
结合和分割。

dispatch_io_create函数生成Dispatch I/O，并指定发生错误时用来执行处理的Block，以及执行该Block的Dispatch
Queue。dispatch_io_set_low_water函数设定一次读取的大小（分割大小），dispatch_io_read函数使用Global
Dispatch Queue开始并列读取。每当各个分割的文件块读取结束时，将含有文件块数据的Dispatch Data传递给dispatch_io_read
函数指定的读取结束是回调用的Block。回调用的Block分析传递过来的Dispatch Data并进行结合处理。如果想提高文件
读取速度，可以尝试使用Dispatch I/O
